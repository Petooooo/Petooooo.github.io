## Goal

The pipeline should help capture work that is easy to lose: deployment changes, useful commands, research notes, and small debugging sessions.

The output is not meant to be published automatically. The useful version is a draft with source metadata, a review checklist, and enough structure that a human can decide whether it is worth keeping.

## Quality Gate

Every generated post starts with `draft: true`, `autoGenerated: true`, `reviewed: false`, source metadata, and a review checklist. Publishing requires an intentional edit.

## Constraints

- No private logs, internal hostnames, tokens, or IP addresses in generated output.
- Fewer drafts are better than noisy activity summaries.
- Source links and dates matter more than confident prose.
- Automation can prepare a note, but it cannot decide that the note is true or useful.
