## Build Strategy

For this site, the clean path is to build Astro once and serve static files. A multi-stage Dockerfile can keep Node out of the runtime image while still making the build repeatable.

```bash
docker compose build
docker compose up -d
```

## Checks To Keep

- `npm run check --prefix apps/web` should catch content schema and Astro type issues.
- `npm run build --prefix apps/web` should produce the same static output used by GitHub Pages.
- Docker builds should not require files outside the repository, except environment variables documented in `.env.example`.

## Deployment Principle

The Raspberry Pi path should be boring: pull from Git, rebuild, restart, and verify. Any state that cannot be recreated from Git should be documented as an external dependency.

## Observation

The most important deployment property for this project is not speed. It is being able to understand the deploy path after leaving it alone for a while.
